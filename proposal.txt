BusYatra Progress Report: What to Include
Guide for Mid-Defense Progress Report Structure
Based on your BusYatra proposal and the sample progress report structure, here is what you should include in each section of your mid-defense progress report.
________________________________________
1. Introduction
What to include: - Very brief recap of proposal context: background, problem definition, objectives, scope/limitations (1–2 paragraphs each, not as long as in the proposal itself) - Emphasize that this report focuses on progress within the 12-week timeline rather than re-justifying the project idea - Mention the submission date and authors
Key points from proposal to reference: - Bus ticketing fragmentation across Trip Turbo, eSewa, Khalti, Sajha Yatayat, FlixBus - The problem: lack of unified, mobile-optimized, real-time seat availability and personalization - Three main objectives: ACID-compliant mobile booking, ML-based seat recommendations, wallet transactions with two-phase commit - Scope: passenger mobile app, operator dashboard, backend services, ML recommendation engine - Limitations: operator participation, geographic coverage, payment gateway integration, etc.
Length: 1.5–2 pages (condensed from the full proposal)
________________________________________
2. Progress Overview (NEW SECTION)
What to include: - A mapping of what was planned in the proposal to your current status - Break down requirements finalization and any changes: - Functional requirements: are they all finalized or did you discover/drop some? - Non-functional requirements: any changes to scalability, security, performance targets? - Feasibility study: confirm if technical, economic, and schedule assumptions held true
•	Module-by-module status (use a table or bulleted list):
–	Mobile App (search, seat layout, booking flow) → Not Started / In Progress / Completed
–	Web Dashboard (operator/admin) → Not Started / In Progress / Completed
–	Backend Services (auth, route, seat, booking, wallet APIs) → Not Started / In Progress / Completed
–	ML Module (KNN recommendation) → Not Started / In Progress / Completed
–	ACID + Optimistic Locking Implementation → Not Started / In Progress / Completed
–	Two-Phase Commit for Wallet → Not Started / In Progress / Completed
–	Testing and Documentation → Not Started / In Progress / Completed
•	Mention any changes from the original objectives or scope
•	Highlight what features/modules will be demo-able at final defense vs. which are prototypes or future work
Length: 1–1.5 pages

3. Methodology and Implementation Status
3.1 Agile Methodology (Updated)
What to include: - Convert the proposal’s generic Agile explanation into what you actually did - Number of sprints/iterations completed so far (e.g., “Completed 4 out of 12 weeks; 2 sprints of 2-3 weeks each”) - What each sprint focused on: - Sprint 1: (e.g., auth, user registration, basic route search) - Sprint 2: (e.g., seat layout UI, booking flow design, database schema refinement) - Sprint 3: (e.g., wallet prototype, KNN baseline model, API integration) - (List any completed sprints)
•	Tools used for agile tracking (Jira, Trello, GitHub Projects, etc.)
•	Iteration cycle: sprint duration, daily standup frequency, review/retrospective format
Length: 0.5–1 page
3.2 Requirements Status
What to include: - State that the functional and non-functional requirements from Section 3.2 of the proposal are now frozen - Highlight any newly discovered requirements during implementation - Call out any requirements that were deprioritized or dropped with justification - Link back to proposal section 3.2.1 and 3.2.2 (functional vs. non-functional)
Length: 0.5 page

Tips for Success
1.	Use Concrete Examples: instead of “we implemented authentication,” say “we implemented JWT-based authentication with role-based access control; users log in with email and receive a token valid for 24 hours.”
2.	Include Screenshots & Diagrams: visual proof that the system is working (app screens, block diagrams, database schema).
3.	Quantify Progress: not “we made good progress on the backend,” but “3 out of 5 core services are complete: auth, route, and booking. Wallet and recommendation services are 50% done.”
4.	Be Honest About Delays: mention challenges and what you learned; judges respect transparency and problem-solving more than pretending everything is perfect.
5.	Link Proposal to Progress: repeatedly reference “as stated in Section 3.3.1 of the proposal, we implemented optimistic locking for seat booking by adding a version field…”
6.	Focus on Demo-Readiness: emphasize what will be live and clickable at mid-defense vs. what is still WIP.
