CHAPTER 1: INTRODUCTION
________________________________________________________________________________

1.1. Introduction
BusYatra is a comprehensive bus ticketing platform designed to digitize and 
centralize the transportation ecosystem in Nepal. It integrates a cross-platform 
mobile application for passengers, a dedicated web dashboard for operators, 
and a robust analytics suite for system administrators.

1.2. Problem Statement
The current bus ticketing landscape in Nepal is heavily fragmented. Passengers 
often navigate multiple platforms (Trip Turbo, Khalti, etc.) with 
inconsistent data, lack of real-time seat tracking, and zero personalization. 
Moreover, operators struggle with manual route management and the high risk of 
double-bookings during peak hours.

1.3. Objectives
- Provide a unified, ACID-compliant mobile booking experience.
- Implement ML-based (KNN) seat recommendations for user personalization.
- Secure wallet-based transactions for reliable bookings.
- Deliver an automated dashboard for real-time fleet analytics.

1.4. Scope and Limitation
- Scope: Passenger app (Booking/Support), Operator app (Route/Fleet Management), 
  Backend API (Services/Analytics).
- Limitations: Depends on internet connectivity; requires operator onboarding 
  for real-time data accuracy; initial ML model accuracy depends on historical 
  data volume.

1.5. Development Methodology
The project followed the **Agile Scrum Methodology**. Sprints were structured in 
2-week intervals, prioritizing the "Minimum Viable Product" (MVP) including 
Authentication and Core Booking flows before moving into advanced features like 
the KNN Recommendation Engine and AI Support Chatbot.

1.6. Report Organization
This report is organized into six chapters: Introduction, Background Study, 
System Analysis, System Design, Implementation and Testing, and Conclusion.

CHAPTER 2: BACKGROUND STUDY AND LITERATURE REVIEW
________________________________________________________________________________

2.1. Background Study
The project utilizes the **MERN Stack** (MongoDB, Express, React, Node.js) 
leveraging its non-blocking I/O model for high concurrency. 
- **React Native (Expo)**: Used for the mobile frontend to ensure a native 
  experience on both Android and iOS.
- **RESTful Architecture**: Ensures stateless communication between the 
  Frontend and the API.
- **KNN (K-Nearest Neighbors)**: A supervised learning algorithm used for 
  demographic-based classification of user seating preferences.

2.2. Literature Review
Analysis of existing systems:
- **Trip Turbo**: Strong mobile presence but lacks AI-driven personalization.
- **Khalti**: Excellent payment gateway but acts as a generalized platform 
  without deep integration into specific bus fleet mechanics.
- **Sajha Yatayat**: Reliable operator service but lacks a sophisticated 
  seat-level availability API for third-party optimization.

3.1. System Analysis
The system focuses on the automation of the entire lifecycle of a bus ticket, 
from route discovery to seat selection and final payment verification.

3.1.1. Requirement Analysis
i. Functional Requirements:
- **Passenger**: Search routes, Select seats, Pay via 
  Khalti/Stripe, Support via Gemini Chatbot.
- **Operator**: Manage assigned vehicles, View booking logs, Update arrival times.
- **Admin**: Approve/Reject operator accounts, Monitor system-wide revenue, 
  Manage dynamic promo codes.

ii. Non-Functional Requirements:
- **Performance**: Response time for seat search < 200ms.
- **Scalability**: Sub-second synchronization across concurrent users.
- **Security**: JWT-based stateless authentication and Bcrypt hashing.

3.1.2. Feasibility Analysis
- **Technical**: MERN stack is modular and well-documented for this domain.
- **Operational**: System is designed with minimalist UI to ensure low learning 
  curves for bus operators.
- **Economic**: Use of free-tier MongoDB Atlas and Vercel reduces initial 
  hosting overhead.
- **Schedule**: The 12-week development timeline prioritized core logic first.

CHAPTER 4: SYSTEM DESIGN
________________________________________________________________________________

4.1. Design (Object Oriented Approach)
- **Database Design**: Implemented as a Document-based model in MongoDB. 
  Normalization is replaced by "Strategic Embedding" (e.g., embedding seat maps 
  within routes for performance).
- **Interface Design**: Mobile app follows a "Tabbed" navigation model (Home, 
  Trips, Support, Profile). Web app uses a sidebar "Dashboard" layout.
- **Deployment**: Architecture is served as a unified Monorepo, split during 
  deployment into Vercel (API/Web) and Expo EAS (Android/iOS).

4.2. Algorithm Details (Core Implementations)
1. **Searching & Array Filtering (JS)**: sub-millisecond route filtering using 
   native JS array reducers.
2. **Balanced Tree (B-Tree) Indexing**: MongoDB-layer optimization for route 
   discovery using logarithmic O(log n) lookup.
3. **Cryptographic Hashing (Bcrypt/SHA-256)**: Secure password salting and 
   payment signature verification for Khalti/Stripe.
4. **Generative AI (Gemini via OpenRouter)**: Context-aware ticketing support 
   Chatbot for 24/7 automated passenger queries.
6. **Seat Integrity**: Validation logic to ensure seats are not double-booked.
7. **KNN Classification**: Euclidean distance calculation for personalization.
8. **Data Serialization (JSON)**: Standardization of cross-component communication.
